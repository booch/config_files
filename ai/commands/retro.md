# Session Retrospective

Analyze our conversation in this session and provide a structured retrospective.

## Analysis Steps

1. **Review the session** — Scan through our conversation to understand what we worked on
2. **Identify patterns** — Look for recurring issues, successful approaches, and friction points
3. **Extract learnings** — What should be remembered for future sessions?

## Retrospective Format

### 🎯 Session Summary

Brief description of what we accomplished (2-3 sentences).

### ✅ What Went Well

- Effective patterns and approaches
- Good collaboration moments
- Things to repeat

### ⚠️ What Could Improve

- Miscommunications or misunderstandings
- Wasted effort or false starts
- Friction points in the workflow

### 💡 Learnings

Concrete things learned that should be remembered:

- Technical discoveries (e.g., "Grammy requires X before Y")
- Workflow insights (e.g., "Always run lint before committing parser changes")
- Project-specific knowledge (e.g., "The AST visitor pattern is in lib/stone/visitor.rb")

### 📝 Proposed Updates

Suggest specific updates to:

1. **AGENTS.md** — Workflow or guideline changes
2. **Skills** — New skills or skill modifications
3. **Commands** — New commands or command modifications
4. **Project docs** — README, architecture docs, etc.

For each proposal, provide:

- The file to update
- The specific change (diff-style if possible)
- Why this change would help

## Output

1. Display the retrospective in the format above
2. Ask: "Would you like me to apply any of these updates?"
3. If approved, use `/learn` to persist the learnings

## Guidelines

- Be specific and actionable, not vague
- Focus on patterns, not individual mistakes
- Prioritize changes that prevent repeated issues
- Keep proposals concise — easy to review and approve
- Don't commit changes without approval
